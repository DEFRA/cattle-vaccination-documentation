# Cattle Vaccination docs site

[Astro](https://astro.build/) + [Starlight](https://starlight.astro.build/) site that publishes the Cattle Vaccination architecture documentation as a static website on GitHub Pages, with interactive [LikeC4](https://likec4.dev/) diagrams embedded in the pages.

The site is built from the same markdown that publishes to Confluence — `technology/` is the single source of truth.

## How it works

```
cattle-vaccination-docs/
├── technology/                       # source of truth (also published to Confluence via mark)
└── astro/
    ├── astro.config.mjs                    # Starlight + LikeC4 vite plugin
    ├── scripts/sync-content.mjs            # technology/ → src/content/docs/
    └── src/
        ├── content.config.ts               # Starlight docs collection
        ├── components/LikeC4View.astro     # <LikeC4View viewId="..." />
        └── content/docs/
            ├── index.mdx                   # landing page (splash + current container view)
            └── technology/           # synced; gitignored, rebuilt by sync script
```

### Content sync

`scripts/sync-content.mjs` runs automatically before `dev` and `build` (both start with `npm run bootstrap`). It reads markdown from `../technology/` and writes it into `src/content/docs/technology/`, transforming Confluence-only syntax along the way:

- Strips Confluence-mark HTML comments — `<!-- Space: -->`, `<!-- Parent: -->`, `<!-- Macro: ... -->`, `<!-- Include: ... -->`.
- Converts `:box:type:title:body:` macros into Starlight admonitions (`:::note[Title] body :::`).
- Converts inline `:status:Name:Colour:Subtle:` macros into Starlight `<Badge text="Name" variant="…" />` (with auto-imported component); colours map Grey→default, Blue→note, Yellow→caution, Green→success, Red→danger.
- Rewrites markdown link targets that point at `.md` source files into Starlight URLs. URL-aware: it resolves the link's target to a source file, computes both the source's and target's destination URL (taking flattening and `README → index` into account), and emits a correctly-relative URL between them. This handles the two off-by-one cases a naive `.md → /` text rewrite would miss: (a) non-README sources, whose URL has an extra trailing-slash directory level (Starlight serves `foo.md` at `/foo/`), and (b) flattened targets, whose URL collapses one path segment (e.g. `sda-adr-006/sda-adr-006.md` → `/technology/.../sda-adr-006/`). Confluence-mark resolves the `.md` form against the source tree; this pass produces Astro-correct URLs in the synced copy, so the same source link works in both targets. External URLs, absolute paths, in-page anchors, and non-markdown relative links are left alone.
- Renames `README.md` → `index.md` so each section gets a proper landing page URL.
- Adds a frontmatter `title` from the first H1 when missing.
- Processes `![alt](path)` image references: if the image basename matches a known LikeC4 view ID (parsed from `.c4` files in the structure-view workspace), the static reference is swapped for an interactive `<LikeC4View viewId="…" />` component and the file is written as `.mdx` with the import auto-prepended; otherwise, if the file exists on disk, the markdown syntax is preserved so Astro's image pipeline optimizes and serves it (typically as a hashed `.webp`); if the file is missing (e.g. `npm run export:structure-view` hasn't been run yet), the reference is replaced with a visible `<span class="missing-diagram">📷 Diagram pending: …</span>` placeholder so the build never fails on un-generated diagrams.
- Copies images alongside their markdown so relative paths keep working.
- **Flattens single-page leaf directories.** A directory that contains exactly one markdown file and has no markdown anywhere deeper is collapsed: `software-architecture/deployment-view/README.md` is written as `software-architecture/deployment-view.md` so the sidebar shows a flat `Software Deployment View` link rather than a kebab-case `deployment-view` group wrapping a single page. Multi-page directories like `scratchpad/` and `interfaces/` are preserved as groups. Image references inside flattened files are rewritten (`images/foo.png` → `deployment-view/images/foo.png`) so they keep resolving against their original on-disk location.

The synced output is gitignored — the build always regenerates it from `technology/`.

### Sidebar

The Starlight sidebar in `astro.config.mjs` mirrors the Confluence Architecture Hub hierarchy: Current State Views, Data Architecture, Integration Architecture, Assurance. Each section uses `autogenerate: { directory: ... }` against the synced tree, so adding a new markdown file under `technology/` automatically gives it a sidebar entry.

**Custom ordering.** Starlight defaults to alphabetical order within an autogenerated section. To override, edit the `SIDEBAR_ORDER` map in `scripts/sync-content.mjs` — keys are source paths under `technology/` (with `/README.md` or `.md` stripped), values are the order number (lower first). The sync injects `sidebar: { order: N, label: "<title>" }` into each page's frontmatter at build time. Pages without an entry fall back to alphabetical. The map lives in the sync (not in source frontmatter) so the source markdown stays clean of Astro-specific metadata and `mark` doesn't have to deal with YAML frontmatter on every file.

**Sub-group labels and positioning.** Starlight's `autogenerate` doesn't read `sidebar.label` or `sidebar.order` from a sub-directory's `index.md` to rename or reposition the **sub-group itself** — it only orders items within the parent group. So a directory called `interfaces/` shows up as a sub-group labelled "interfaces" (lowercase, from the directory name), positioned alphabetically. To work around this, configure the *parent* section manually in `astro.config.mjs`: list its top-level items explicitly, and use a nested `autogenerate` only for sub-groups whose internal order you want managed via the SIDEBAR_ORDER map. See the Integration Architecture section in `astro.config.mjs` for the pattern.

### Mermaid diagrams

Fenced `mermaid` code blocks render client-side. The remark plugin at `src/lib/remark-mermaid.mjs` converts each `mermaid` block into a `<pre class="mermaid">` element (bypassing Shiki, which would otherwise wrap each token in a `<span>` and break mermaid's parser). A small init script declared in `astro.config.mjs` loads mermaid.js from `esm.sh` and calls `mermaid.run({ querySelector: ".mermaid" })` once the DOM is ready. No npm dep, no build-time browser. To change the theme or default config, edit `mermaidInitScript` at the top of `astro.config.mjs`.

### Interactive LikeC4 diagrams

The LikeC4 Vite plugin reads `.c4` sources from `../technology/software-architecture/structure-view`. Embed any view by ID:

```mdx
import LikeC4View from "../../components/LikeC4View.astro";

<LikeC4View viewId="cvac_domain_current_container" />
```

**Auto-swap from static images.** You usually don't need to write the component manually. The PNG export uses `<viewId>.png` as the filename, so any markdown image reference whose basename matches a known view ID is swapped for `<LikeC4View>` at sync time. The same source file then renders static PNGs in Confluence and interactive views on the Astro site, with no per-target authoring. Files that get an injected component are written as `.mdx` (with the import added automatically) and have HTML comments converted to MDX expression comments so Marp directives don't break the parser.

### Tabs

Wrap a sequence of headings in `<!-- Tabs:start -->` / `<!-- Tabs:end -->` to render them as Starlight tabs in Astro. Each heading becomes a tab label; the content between consecutive headings becomes the tab body. Confluence ignores the markers and renders the headings normally, so the same source serves both targets.

```md
<!-- Tabs:start -->

### Current

![Domain (current)](images/cvac_domain_current_container.png)

### Future

![Domain (future)](images/cvac_domain_current_context.png)

<!-- Tabs:end -->
```

The sync auto-imports `Tabs` and `TabItem` from `@astrojs/starlight/components` and writes the file as `.mdx`. Tabs processing runs after the LikeC4 view auto-swap, so static-image references inside the tabs get upgraded to interactive views automatically.

To see the available view IDs:

```sh
grep -h "^[[:space:]]*view " ../technology/software-architecture/structure-view/**/*.c4 \
                              ../technology/software-architecture/structure-view/*.c4 \
  | awk '{print $2}' | sort -u
```

## Local development

From the repo root:

```sh
npm run site:dev       # http://localhost:4321/cattle-vaccination-docs/
npm run site:build     # produces astro/dist/
npm run site:preview   # serve the built site locally
```

`site:dev` and `site:build` are self-bootstrapping — `dev` and `build` both start with `npm run bootstrap`, which runs `npm install && npm run sync && node scripts/build-likec4.mjs` inside `astro/`, so a freshly cloned repo works without an explicit install step. (This is chained explicitly rather than using `predev` / `prebuild` hooks because npm skips pre/post lifecycle scripts when `ignore-scripts=true`.) Use `npm run site:install` if you want to warm the cache up front (e.g. on CI before a separate build step).

Or from inside `astro/`:

```sh
npm install
npm run dev
npm run build
```

## Deployment

`.github/workflows/ci.yaml` runs on push to `main`:

1. `npm ci` in `astro/`
2. `npm run sync` (regenerates synced content from `technology/`)
3. `npm run build`
4. Uploads `astro/dist/` and deploys via `actions/deploy-pages`

### One-time GitHub setup

In the repo settings:

- **Settings → Pages → Build and deployment → Source: GitHub Actions**

The workflow respects two optional repo-level settings:

| Setting | Where | Purpose |
| --- | --- | --- |
| `PAGES_SITE_URL` | repo variable | Custom/private Pages URL. When set, the site builds with `base: "/"`. Leave unset for the default `https://defra.github.io/<repo>/`. |
| (none) | — | The default deploy URL is derived from `BASE_PATH=/${repo-name}`. |

After the first successful run, the site is live at the URL printed in the deploy job (or your `PAGES_SITE_URL`).

## Adding a new page

Add a markdown file anywhere under `technology/` — the next `npm run site:dev` or `site:build` picks it up automatically. To embed an interactive diagram, save the file as `.mdx` instead of `.md` and use the `<LikeC4View>` component shown above.

## Linking between pages

Markdown links inside the body — e.g. `[See more](/technology/software-architecture/)` — are auto-rewritten by Starlight to include the configured `base` path. **Hero action links in frontmatter are not** — if you add hero actions on a splash page, use a `<LinkButton>` in the MDX body with `${import.meta.env.BASE_URL}` instead, as `index.mdx` does.

## Things to know

- **No Confluence publishing here** — that lives in the root `npm run confluence:publish` script (using `mark`). This site is GitHub Pages only.
- **No PNG export here either** — the existing `npm run export:structure-view` continues to handle static PNGs for slides and Confluence; the Astro site shows the *interactive* LikeC4 views directly.
